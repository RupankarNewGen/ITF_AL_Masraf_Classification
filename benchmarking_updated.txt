for benchmarking the IDP (Intelligent Document Processing) system against
trade-finance document images. Both pipelines ultimately measure how accurately
the system classifies and extracts fields from multi-page document workitems.

1.  OVERVIEW OF THE TWO PIPELINES
================================================================================

Pipeline A — "Full Inference & Extraction Benchmarking"
    Designed to process RAW IMAGE FOLDERS through the complete IDP system,
    from image pre-processing and inference all the way through field-level
    extraction accuracy reporting.  It is used when you START from scratch
    (no prior JSON results) and want to run a fresh end-to-end benchmark.

    Files involved:
        (1) automate_phase_1.py   → Image preparation (annotation-folder scan)
        (2) automate_phase_2.py   → Inference + extraction evaluation

Pipeline B — "Split-Workitem Classification & Extraction Benchmarking"
    Designed to process EXISTING JSON RESULT FILES that contain multi-page
    workitem inference outputs.  It SPLITS them into individual page-level
    JSONs, classifies each page against a known ground truth, and then runs
    extraction benchmarking only on correctly-classified pages.

    Files involved:
        (1) split_workitem_jsons.py         → Splits master JSONs → page JSONs
        (2) evaluate_split_jsons.py         → Classifies & evaluates accuracy
        (3) automate_benchmarking_split.py  → Runs full extraction benchmark
                                              on correctly-classified pages


================================================================================
2.  PIPELINE A — FULL INFERENCE & EXTRACTION BENCHMARKING
================================================================================

2.1  USE CASE
─────────────
    Use Pipeline A when:
      • You have a list of raw image annotation directories (stored in
        FINAL_MATCHED_DIRECTORIES.json) and want to benchmark from images.
      • You want to run the full IDP API inference pipeline fresh.
      • You need a single command to orchestrate image gathering, inference,
        YOLO label conversion, and extraction report generation from scratch.

    This pipeline is the "offline training/QA" benchmark: you point it at
    annotation folders, run it, and get class-wise extraction accuracy numbers.

─────────────────────────────────────────────────────────────────────────────
2.2  PHASE 1: automate_phase_1.py
─────────────────────────────────────────────────────────────────────────────
File Path:
    /datadrive2/IDF_AL_MASRAF/ITF_Al_Masraf_Classification/automate_phase_1.py

Purpose:
    Iterates over a list of annotation directories and, for each directory,
    invokes the helper script `get_images_from_annoation_folders.py` to
    extract and organise the relevant document images into a centralised
    inference-ready folder structure.

Entry Point:
    python automate_phase_1.py

Key Configuration Constants (top of file):
  ┌─────────────────────────────────────┬──────────────────────────────────────────────────────────────────┐
  │ Constant                            │ Description                                                      │
  ├─────────────────────────────────────┼──────────────────────────────────────────────────────────────────┤
  │ FINAL_MATCHED_DIRECTORIES_JSON      │ Path to JSON list of annotation directories to process.          │
  │                                     │ Each entry is a string path to a source annotation folder.       │
  │ GET_IMAGES_SCRIPT                   │ Path to the helper script that performs the actual image pull.   │
  └─────────────────────────────────────┴──────────────────────────────────────────────────────────────────┘

Execution Logic:
    1. Reads FINAL_MATCHED_DIRECTORIES.json — expects a JSON array of directory paths.
    2. For each directory, calls `get_images_from_annoation_folders.py` with
       argument `--directory <dir>` using the qwen_env Python interpreter.
    3. Any CalledProcessError per directory is caught and logged without
       stopping the entire run.
    4. Prints a final success checkpoint on completion.

Interpreter Used:
    /datadrive2/IDF_AL_MASRAF/qwen_env/bin/python

Expected Input:
    FINAL_MATCHED_DIRECTORIES.json — example structure:
        [
            "/datadrive2/.../BOL_batch1",
            "/datadrive2/.../CI_batch2"
        ]

    • Images from each annotation folder are organised under:
      Data/Raw_ouput_data/images_for_inference/<folder_name>/<class_name>/

2.3  PHASE 2: automate_phase_2.py
─────────────────────────────────────────────────────────────────────────────
File Path:
    /datadrive2/IDF_AL_MASRAF/ITF_Al_Masraf_Classification/automate_phase_2.py

Purpose:
    Orchestrates the core benchmarking workflow after Phase 1. Sequentially:
      - Runs the IDP inference API pipeline on prepared images.
      - Reads per-folder accuracy reports.
      - Qualifies directories that pass an 80% inference threshold.
      - For qualified directories, runs the benchmarking data constructor,
        YOLO-to-GroundTruth converter, and extraction report generator for
        each document class (BOE, BOL, CI, COO, CS, PL).
      - Saves a consolidated JSON tracking file of all results.

Entry Point:
    python automate_phase_2.py

STEP-BY-STEP EXECUTION:

    STEP 1 — API Inference Pipeline
        Runs `idp_full_pipeline_api_request_nested.py` which processes all
        images for inference, depositing result JSONs and report TXT files
        under Data/Raw_ouput_data/images_for_inference_results/.

    STEP 2 — Qualify & Process Results
        For each sub-folder in images_for_inference_results/:
          a) Reads `report_<folder_name>.txt` and extracts the "Accuracy: XX%"
             line using regex.
          b) If accuracy >= 80.0%:
               - Marks folder as qualified.
               - Runs `benchmarking_data_constructor.py` to build class-level
                 ground-truth data structure in Working_Data/<folder_name>/.
               - For each class directory (BOE, BOL, CI, COO, CS, PL):
                   i)  Runs `yolo_to_gt_converter.py` to convert YOLO annotation
                       labels to the ground-truth format expected by the evaluator.
                   ii) Runs `report_generator_rupankar_v3.py` to produce the
                       field-level extraction accuracy report.
                   iii) Reads the `final_report_*.txt` (JSON format) from the
                        latest `label_wise` sub-directory to extract
                        OVERALL.Complete_Match_Percentage.
          c) If accuracy < 80.0%: the folder is skipped (disqualified).

    STEP 3 — Save Tracking JSON
        Writes benchmarking_final_results.json to Working_Data/ with structure:
            {
                "<folder_name>": {
                    "Overall_Inference_Accuracy": <float>,
                    "Classes": {
                        "<class_code>": <extraction_accuracy_float>,
                        ...
                    }
                }
            }

3.  PIPELINE B — SPLIT-WORKITEM CLASSIFICATION & EXTRACTION BENCHMARKING
================================================================================

3.1  USE CASE
─────────────
    Use Pipeline B when:
      • You already have multi-page workitem JSON files from a prior inference
        run (stored in Data/Test_jsons/) and want to evaluate them WITHOUT
        running inference again.
      • You need page-level classification accuracy across workitems.
      • You want to run extraction benchmarking ONLY on images where the
        model's classification was verified correct by ground truth.
      • You are working with pre-split or mixed document batches (test sets)
        rather than single-class annotation directories.

    This pipeline is the "post-inference evaluation" benchmark: you have
    JSON results, now you want to know how good they are at both the
    classification AND the extraction level.

─────────────────────────────────────────────────────────────────────────────
3.2  STEP 1: split_workitem_jsons.py
─────────────────────────────────────────────────────────────────────────────
File Path:
    /datadrive2/IDF_AL_MASRAF/ITF_Al_Masraf_Classification/split_workitem_jsons.py

Purpose:
    Explodes master (multi-page) workitem JSON files into individual
    single-page JSON files, one per document page. The output format is
    compatible with the evaluation and benchmarking scripts that follow.

Entry Point:
    python split_workitem_jsons.py

Input Format (master JSON):
    Each JSON in Data/Test_jsons/ follows a structure like:
        {
            "result": {
                "input_tif_file__0.png": { <page_0_inference_data> },
                "input_tif_file__1.png": { <page_1_inference_data> },
                ...
            }
        }
    OR (alternate top-level format without a "result" key):
        {
            "input_tif_file__0.png": { ... },
            "input_tif_file__1.png": { ... }
        }

Execution Logic:
    1. Scans DATA/Test_jsons/ for all *.json files.
    2. For each master JSON:
       a) Reads the "result" dict (or falls back to top-level keys).
       b) Creates a subdirectory in Test_jsons_splitted/<workitem_name>/.
       c) For each page key (e.g. `input_tif_file__2.png`):
          - Extracts the zero-based page index from the `__<N>` suffix via regex.
          - Converts to 1-based page number.
          - Writes a single-page JSON named: `<workitem>_page_<N>.json`
            with content:
                { "<workitem>_page_<N>.jpeg": { <page_inference_data> } }
    3. Reports total files processed and total single-page JSONs generated.

Key Configuration:
  ┌────────────┬─────────────────────────────────────────────────────────────┐
  │ Variable   │ Description                                                 │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ INPUT_DIR  │ Source folder containing master multi-page workitem JSONs.  │
  │ OUTPUT_DIR │ Destination for the split single-page JSON sub-folders.     │
  └────────────┴─────────────────────────────────────────────────────────────┘

Output Structure:
    Data/Test_jsons_splitted/
        <workitem_name>/
            <workitem_name>_page_1.json
            <workitem_name>_page_2.json
            ...

─────────────────────────────────────────────────────────────────────────────
3.3  STEP 2: evaluate_split_jsons.py
─────────────────────────────────────────────────────────────────────────────
File Path:
    /datadrive2/IDF_AL_MASRAF/ITF_Al_Masraf_Classification/evaluate_split_jsons.py

Purpose:
    Evaluates the classification accuracy of each single-page JSON produced
    by Step 1 by comparing its `predicted_class` against a ground truth
    label learned from the physical image directory structure.

Entry Point:
    python evaluate_split_jsons.py

GROUND TRUTH BUILDING (build_ground_truth_map):
    The script learns the ground truth label for each image by scanning a
    predefined list of "target search folders" (TARGET_SEARCH_FOLDERS).
    Each target folder is expected to have sub-directories named after
    document classes (e.g. "Bill_of_Lading", "BOL", "Commercial_Invoice"):

        test_set2/
            Bill_of_Lading/
                doc001_page_1.jpeg   → GT = "BOL"
                doc001_page_2.jpeg   → GT = "BOL"
            Commercial_Invoice/
                doc007_page_1.jpeg   → GT = "CI"

    All image filenames (without extension) are indexed to their standard
    class code using the CLASS_MAPPING dictionary.

EVALUATION LOGIC (main):
    For each workitem directory in Test_jsons_splitted/:
       1. For each page JSON file:
          a) Derives the image base name from the JSON filename.
          b) Looks up the GT class from the image index.
             - If found and class != "Others": records a proper GT label.
             - If found and class == "Others": records as "Mapped_To_Others".
             - If NOT found: defaults GT to "Others" and logs to
               "Not_Found_Defaulted_To_Others".
          c) Reads `predicted_class` from the JSON file.
          d) Compares GT (case-insensitive) to predicted.
          e) Writes one row to evaluation_report.csv per page.
       2. Prints per-workitem accuracy.
    3. Saves mapping_diagnostics.json in SPLIT_RESULTS_DIR with three keys:
          - "Mapped_To_Class":               dict of class → [image_names]
          - "Mapped_To_Others_From_Folder":  list of images found in "Others" dir
          - "Not_Found_Defaulted_To_Others": list of images with no GT match

Key Configuration:
  ┌──────────────────────┬──────────────────────────────────────────────────────────────┐
  │ Constant             │ Description                                                  │
  ├──────────────────────┼──────────────────────────────────────────────────────────────┤
  │ TARGET_SEARCH_FOLDERS│ List of root folders that contain class-named sub-dirs       │
  │                      │ where ground truth images physically reside.                 │
  │ SPLIT_RESULTS_DIR    │ Root of the split JSONs (output of Step 1).                 │
  │ CLASS_MAPPING        │ Dict normalising folder names → standard class codes.        │
  │ VALID_EXTS           │ Set of image file extensions considered for GT indexing.     │
  └──────────────────────┴──────────────────────────────────────────────────────────────┘

Output Per Workitem:
    Data/Test_jsons_splitted/<workitem_name>/evaluation_report.csv
        Columns: Image Name | Ground Truth Class | Predicted Class | Is Correct

Global Output:
    Data/Test_jsons_splitted/mapping_diagnostics.json
        Used to diagnose any GT lookup failures (defaulted-to-Others images).

IMPORTANT NOTE ON GT MAPPING:
    If a large fraction of images fall into "Not_Found_Defaulted_To_Others",
    it likely means:
      - The image filenames in the JSON do not exactly match the filenames
        present in TARGET_SEARCH_FOLDERS.
      - A required test folder path is missing from TARGET_SEARCH_FOLDERS.
    Always inspect mapping_diagnostics.json after running this step.

3.4  STEP 3: automate_benchmarking_split.py
─────────────────────────────────────────────────────────────────────────────
File Path:
    /datadrive2/IDF_AL_MASRAF/ITF_Al_Masraf_Classification/automate_benchmarking_split.py

Purpose:
    The orchestration script for extraction benchmarking on split workitems.
    It reads the evaluation_report.csv from Step 2, selects only correctly-
    classified non-"Others" pages, then runs the full extraction benchmarking
    pipeline (constructor → YOLO converter → report generator) on those pages,
    and finally aggregates results into page-wise and document-level CSVs.

Entry Point:
    python automate_benchmarking_split.py

PRE-REQUISITES:
    - Step 1 (split_workitem_jsons.py) must be complete.
    - Step 2 (evaluate_split_jsons.py) must be complete so that
      evaluation_report.csv exists in each workitem sub-directory.

IMAGE INDEXING (build_image_index):
    Before processing workitems, builds a global index of all physical image
    files found in TARGET_SEARCH_FOLDERS (the same folders as Step 2).
    The index maps: image_base_name → absolute_image_file_path.
    This index is later used to locate and COPY real images into the local
    `images/` directory needed by the YOLO-to-GT converter.

MAIN PROCESSING LOOP:
    For each workitem with an evaluation_report.csv:
        1. Filter rows where GT != "Others" AND Is_Correct == "TRUE".
           → These are the "valid images" that passed classification.
        2. Skip workitems with zero valid images.
        3. Create a DUMMY REPORT TXT:
           A synthetic report file listing each valid image as correctly
           predicted. Format: "[✓] <img_name>.jpeg: Pred=<GT> | GT=<GT>"
           This tricks `benchmarking_data_constructor.py` into treating
           all listed images as 100% accurate, thus building the full GT
           data structure for them.
        4. Run `benchmarking_data_constructor.py`:
           Builds the class-specific directory structure under:
               Data/Test_data_Splitted_Results/<workitem>/<class_name>/
                   labels/      — YOLO .txt annotation files
        5. For each class (BOE/BOL/CI/COO/CS/PL) found:
           a) Create a local `images/` sub-directory.
           b) Copy the relevant physical images (from image index) into it.
           c) Run `yolo_to_gt_converter.py` to convert labels to GT format,
              outputting to `New_Master_Data_Merged/`.
           d) Run `report_generator_rupankar_v3.py` to generate extraction
              accuracy CSVs and the final report JSON.
        6. AGGREGATION — build_page_wise_summary (called twice):
           a) Native pipeline CSVs:
              Pattern: **/result_path/*/*_analysis_pre_valid_after_fuzzy_
                       match_post-processing_latest.csv
              Output: <workitem>_final_document_page_wise_summary.csv
           b) Strict 75% fuzzy CSVs:
              Pattern: **/result_path/*/*_analysis_fuzzy_75.csv
              Output: <workitem>_final_document_page_wise_summary_75.csv

AGGREGATION LOGIC (build_page_wise_summary):
    - Reads all matching CSVs across a workitem's class directories.
    - Tags each row with the class name (extracted from CSV filename prefix).
    - Groups by File_Name (one row per page image).
    - For each page: filters out rows where 'actual' (GT value) is empty/NaN.
    - Computes: Accuracy_Percentage = matched_fields / total_fields * 100
    - Appends an OVERALL_DOCUMENT row with aggregate stats for the workitem.
    - Writes a final summary CSV.

Key Configuration:
  ┌─────────────────────────┬──────────────────────────────────────────────────────────────┐
  │ Constant                │ Description                                                  │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────┤
  │ BASE_DIR                │ Project root.                                                │
  │ SPLIT_RESULTS_DIR       │ Directory of split JSONs + evaluation_report.csv files.      │
  │ WORKING_DATA_DIR        │ Output root for extraction benchmarking results.             │
  │ CONSTRUCTOR_SCRIPT      │ benchmarking_data_constructor.py path.                       │
  │ YOLO_GT_SCRIPT          │ yolo_to_gt_converter.py path.                               │
  │ REPORT_EVAL_SCRIPT      │ report_generator_rupankar_v3.py path.                       │
  │ CLASSES_FILE_MAP        │ Dict of class code → classes.txt path (same as Pipeline A). │
  │ TARGET_SEARCH_FOLDERS   │ Image source folders used to build physical image index.    │
  └─────────────────────────┴──────────────────────────────────────────────────────────────┘

4.1  PURPOSE & ROLE IN BENCHMARKING
─────────────────────────────────────
    This is a third-party library file from the TorchMetrics package
    (Apache 2.0 License). It provides GPU/CPU-compatible, distributed-safe
    metric aggregation classes used internally by the evaluation pipeline's
    metric computation modules (e.g. in report generators that rely on
    PyTorch-based scoring).

    In the context of this project, torchmetrics aggregation classes may be
    used in the model evaluation codebase to:
      - Accumulate match/no-match counts across document batches.
      - Compute running means of extraction accuracy scores without storing
        all intermediate values in memory.
      - Handle NaN values in extracted field comparisons gracefully.

4.2  KEY CLASSES
─────────────────
    ┌────────────────┬──────────────────────────────────────────────────────────┐
    │ Class          │ Description                                              │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ BaseAggregator │ Abstract base class. Handles NaN strategy (error/warn/   │
    │                │ ignore/disable/float imputation). All subclasses inherit │
    │                │ _cast_and_nan_check_input() for safe tensor conversion.  │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ MaxMetric      │ Tracks the running maximum of a value stream.            │
    │                │ State: max_value (scalar). update() calls torch.max().   │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ MinMetric      │ Tracks the running minimum of a value stream.            │
    │                │ State: min_value (scalar). update() calls torch.min().   │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ SumMetric      │ Accumulates the sum of all values seen.                  │
    │                │ State: sum_value (scalar). update() calls .sum().        │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ CatMetric      │ Concatenates all value tensors into a single tensor.     │
    │                │ State: value (list → tensor at compute time).            │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ MeanMetric     │ Computes weighted or unweighted running mean.            │
    │                │ State: mean_value (sum of value*weight), weight (sum).   │
    │                │ compute() = mean_value / weight.                         │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ RunningMean    │ MeanMetric over a sliding window of N values.            │
    │                │ Useful for per-batch accuracy smoothing during training. │
    ├────────────────┼──────────────────────────────────────────────────────────┤
    │ RunningSum     │ SumMetric over a sliding window of N values.             │
    └────────────────┴──────────────────────────────────────────────────────────┘

    NaN Strategy Options (all classes):
      - 'error'   : raise RuntimeError if NaN detected
      - 'warn'    : log a warning and silently remove NaN entries (default)
      - 'ignore'  : silently remove NaN entries
      - 'disable' : skip NaN checks entirely
      - <float>   : impute NaN values with the given float

7.  TROUBLESHOOTING & FAQs
================================================================================

Q1: A large number of images are defaulting to "Others" in Step 2/Pipeline B.
    → Inspect Data/Test_jsons_splitted/mapping_diagnostics.json.
    → Check the "Not_Found_Defaulted_To_Others" list.
    → Verify that the image base names in the list match actual files in
      TARGET_SEARCH_FOLDERS. Mismatches in naming convention (e.g.
      page_1 vs page_01) will cause lookup failures.
    → Add the correct source folder to TARGET_SEARCH_FOLDERS in both
      evaluate_split_jsons.py and automate_benchmarking_split.py.

Q2: Phase 2 / Pipeline A: a folder is disqualified even though it looks correct.
    → Check the `report_<folder>.txt` file in images_for_inference_results/.
    → Ensure it contains a line matching "Accuracy: XX.XX%".
    → The threshold is 80.0%. Edit `if accuracy >= 80.0:` in automate_phase_2.py
      if you need to adjust it.

Q3: The YOLO converter fails with a missing images directory.
    → In Pipeline B, physical images are copied from TARGET_SEARCH_FOLDERS
      into a local `images/` dir using the image index. If build_image_index()
      returns 0 images, TARGET_SEARCH_FOLDERS is likely empty or incorrect.
    → In Pipeline A, the img_dir is expected at:
      images_for_inference/<folder_name>/<class_name>/. Run Phase 1 first.

Q4: How do I add a new document class (e.g. "LC")?
    → Create a classes.txt annotation file for the new class.
    → Add it to CLASSES_FILE_MAP in both automate_phase_2.py and
      automate_benchmarking_split.py.
    → Add the class code and folder name to CLASS_MAPPING in
      evaluate_split_jsons.py.
    → Ensure source images for the new class exist in a TARGET_SEARCH_FOLDER
      under a sub-directory named after the class (e.g. "Letter_of_Credit/").

Q5: Can Pipeline A and Pipeline B be run independently?
    → Yes. They share no runtime dependencies on each other's output.
      They share only static configuration (CLASSES_FILE_MAP, scripts paths).

Q6: What does the 75% fuzzy CSV represent vs the native pipeline CSV?
    → The native pipeline CSV uses the default fuzzy match threshold of the
      report generator (typically > 75% string similarity for a "match").
    → The _fuzzy_75.csv enforces a strict 75% uniform threshold across all
      field types, providing a more conservative accuracy estimate.

Purpose:
    Converts YOLO bounding-box annotation files (.txt) into a ground-truth
    JSON format that the extraction report generators can consume.
    It cross-references each YOLO box against pre-computed OCR word coordinates
    to extract the ground-truth text for each labelled field.

Core Function: process_file(yolo_file_path, ocr_file_path, image_path, classes_map, output_file_path)

INTERNAL LOGIC:
    For each YOLO .txt file in yolo_dir:
      1. Reads the image dimensions (using PIL) to de-normalise YOLO coords.
      2. Reads the OCR file (at ocr_dir/<base_name>_textAndCoordinates.txt).
         The OCR file is a Python-literal dict with key "ocrContent": a list of
         word dicts each containing keys: word, x1, y1, x2, y2.
      3. For each YOLO bounding box:
         a) De-normalises (cx, cy, w, h) → absolute pixel (x1, y1, x2, y2).
         b) Maps class_idx → field_name using the classes.txt file.
         c) Finds all OCR words where the word's bounding box overlaps the YOLO
            box by >= 50% (intersection / ocr_word_area >= 0.5).
         d) Sorts matched words by (row, column) — row tolerance = 10 pixels.
         e) Joins word texts into a single string → the GT text value.
      4. Writes a JSON output file: <base_name>_labels.txt
         Format: { "<field_name>": [["<gt_text>", [x1,y1,x2,y2]], ...], ... }

CLI Arguments:
    --yolo_dir      (required) Directory containing YOLO .txt label files.
    --img_dir       (required) Directory containing corresponding images.
    --classes_file  (required) Path to classes.txt mapping index → field name.
    --output_dir    (required) Where to write the converted GT JSON files.
    --ocr_dir       (optional) OCR data directory. Default:
                    Data/ocr_data/

Output:
    output_dir/<image_base_name>_labels.txt
    (one JSON file per YOLO label file, containing all field → GT-text mappings)

Notes:
    • Skips any image for which the OCR file or image file is missing.
    • The output format is consumed by report_generator_rupankar_v3.py as the
      "ground truth data" (New_Master_Data_Merged/).

        Extracts extraction_result, sign/stamp detections from prediction.
        Calls generate_accuracy_data() to compare each field's predicted vs
        GT value, applying fuzzy matching and post-processing rules.

    generate_reports():
        Calls all three report generators:
          1. label_wise_report()        → all labels, no filtering
          2. rules_required_keys_report() → only mandatory fields
          3. most_occurred_label_report() → only high-frequency fields

Config Files Read (from script directory):
    config.ini    — Contains paths to pp_json (post-processing rules) per
                    product/document code combination.
    The post-processing JSON defines fuzzy match strategies for each field.

Output (written by prepare_report() / post_process_rupankar_v2):
    folder_path/result_path/label_wise/<doc_code>_<timestamp>/
        *_analysis_pre_valid_after_fuzzy_match_post-processing_latest.csv
        *_analysis_fuzzy_75.csv
        final_report_<timestamp>.txt   ← JSON with OVERALL.Complete_Match_Percentage
